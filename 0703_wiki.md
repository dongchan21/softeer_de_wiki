* ETL 구조를 어떻게 나눌지 고민했다. 처음에는 모든 로직을 한 함수에 작성했지만, Extract, Transform, Load를 각각 함수로 분리하고 run_etl()에서 전체 흐름을 관리하도록 변경했다. 덕분에 각 단계의 역할이 명확해졌고 유지보수와 재사용성이 좋아진다는 점을 배웠다.
* 데이터가 갱신되어도 재사용 가능한 코드를 만드는 방법을 고민했다. 처음에는 "IMF (2026)[1]"처럼 컬럼명을 직접 사용했지만, 연도가 바뀌면 코드가 동작하지 않는다는 점을 깨달았다. 이후 "IMF"가 포함된 컬럼을 동적으로 찾는 방식으로 수정해 변화에 대응할 수 있도록 개선했다.
* 원하는 테이블만 안정적으로 추출하는 방법을 고민했다. 단순히 첫 번째 테이블을 가져오는 대신 caption을 기준으로 "GDP forecast or estimate" 테이블을 찾도록 구현했다. 이를 통해 웹페이지 구조가 일부 변경되더라도 원하는 데이터를 더 안정적으로 가져올 수 있다는 점을 배웠다.
* 지역(Region) 정보를 어떻게 추가할지 고민했다. 처음에는 대륙별 GDP 페이지를 추가로 크롤링하는 방법을 생각했지만, 구현이 복잡해질 수 있다고 판단했다. 대신 country_converter 라이브러리를 사용해 국가명을 기준으로 대륙 정보를 자동으로 매핑하는 방식을 선택했고, 기존 ETL 구조를 크게 변경하지 않고도 기능을 구현할 수 있었다.
* 데이터 저장 방식을 고민했다. JSON 파일 저장뿐 아니라 SQLite 데이터베이스 저장 방식도 구현해 보면서, 같은 ETL 구조에서도 Load 단계만 변경하면 다양한 저장소를 지원할 수 있다는 점을 이해했다. ETL에서 각 단계를 분리하는 이유를 직접 체감할 수 있었다.
* 과거 데이터 관리의 필요성에 대해 생각해봤다. 처음에는 최신 데이터만 저장하면 된다고 생각했지만, 연도별 GDP 변화를 조회하려면 데이터를 누적 관리해야 한다는 점을 알게 되었다. ETL은 단순히 데이터를 가져오는 것뿐 아니라, 이후 어떻게 활용할지를 고려해 저장 구조를 설계하는 것도 중요하다는 점을 배웠다.
